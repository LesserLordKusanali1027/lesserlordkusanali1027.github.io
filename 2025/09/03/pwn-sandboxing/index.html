Sandboxing (沙箱)

发表于2025-09-03|Pwn.college
|浏览量:

沙箱技术，通过隔离运行环境防止对系统其他部分造成破坏，虚拟机、容器等都属于沙箱技术，下面将介绍几个简单的沙箱技术。

Filesystem Sandboxing

介绍chroot：

一种隔离文件系统访问的沙箱技术，将进程的根目录修改到一个子目录下，这样该进程就无法访问除子目录以外的其他地方。

比如输入下面的命令，会运行/tmp/jail/busybox，但/busybox眼中的根目录/实际上是系统中的/tmp/jail；而如果程序busybox在/tmp/jail下运行cd ..，还会停留在/tmp/jail中，因为这在busybox看来就是根目录了。

chroot实现方法：

事实上chroot只做了：

  1. 对该进程而言，/表示/tmp/jail
  2. 对该进程而言，/tmp/jail/..会进入/tmp/jail

chroot并没有：

  1. 真正对进程关闭其他文件的权限
  2. 主动进入jail目录下
  3. …

例如，下面是一个正确的程序，/busybox在新的根目录下开始运行。

但如果忘记了chdir()，/busybox运行在/tmp/jail之外，虽然/仍表示根目录/tmp/jail，但其他文件可以用相对路径访问。

chroot漏洞1：

chroot不会检测之前打开的资源，如果chroot前留下了jail外面的文件描述符，那么在jail里可通过openat和execveat联系外部。

chroot漏洞2：

单个进程可以多次chroot，后面的会把前面的覆盖。

总结：

chroot有很多方法可以绕过；而且没有提供其他的隔离，比如进程PID、网络、进程通信。

现在采用了其他替代方案，如cgroups、namespaces、seccomp，会在后续讨论。

System Call Sandboxing

介绍seccomp：

用于限制系统调用，内核级别，可以用seccomp来禁用或允许特定的系统调用，是现代沙箱的重要一部分，Linux、Chrome、Firefox都依赖它来保证安全性。

逃离seccomp 1： permissive policies

系统调用数量太多，难以用seccomp实现完美管理；继承关系很复杂，开发者可能会选择宽松的规则以避免某些系统调用在之后被禁止。

一个典型的例子是如果seccomp允许ptrace，那么内存、寄存器都可随意修改，可以跳转到另一个进程中实现沙箱逃逸，这在CTF中比较常见。

还比如sendmsg除了可以在进程间发送消息，还可以把消息伪造成文件；prctl可以造成很大的影响；process_vm_writev可以取得其他进程的一些权限，都可以用来逃出沙箱。

这些系统调用很难被完美地管理，一旦有所疏漏就可以利用。

Sandboxing via Namespaces

Namespace（命名空间）可以提供操作系统级别的隔离，是实现Docker、Cloud的关键技术，命名空间其实是一系列资源的集合，包括网络设备、挂载点、PID等。

先介绍挂载(mount)的概念：

Linux系统把设备等关联到文件系统中，这些文件就成为访问这些设备的入口，比如键盘、鼠标、磁盘分区、U盘等在文件系统中都有对应的文件，这个关联过程就叫做挂载(mount)，

比如/dev/input/mouse0对应鼠标，执行cat /dev/input/mouse0后移动鼠标就可以看到输出。

而运行mount可以看到文件系统的挂载情况，下面截取了4行，/dev文件夹中挂载了所有连接到电脑的设备，前面的/dev/input/mouse0就连接了鼠标。

介绍unshare：

正常的fork()得到的子进程会与父进程共享很多资源，而通过unshare可以在新的namespace中开启进程，使子进程使用独立的资源。

使用unshare创建新进程，-m表示开启的新shell将使用新的挂载

这时运行mount其实还能看到父级的挂载

执行mount进行新的挂载，将/bin绑定到./tmp_bin，现在对./tmp_bin操作就相当于对/bin。使用umount ./tmp_bin可以解除挂载，文件夹原本的内容重新可见。

但是在另一个shell中查看tmp_bin时，并不能看到/bin的内容，这是因为两个shell位于不同的mount namespace中，挂载(mount)操作是相互独立的。

不过两个shell都可以看到tmp_bin，因为文件系统是共享的，只是挂载的方式不同。
