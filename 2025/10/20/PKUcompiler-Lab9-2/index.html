Lab9+2，结束

前面所做的一切，不过是在 AST 上走来走去，是只要花时间就能做出来的大一点的垃圾；而这最后要做的优化和寄存器分配，才是我真正期待的事情，是我费力写完前面一坨的终极目标，是可以让我的编译器从垃圾堆中脱颖而出的一抹光。

概述

优化，按照优化的时机/优化的对象，大致可分为两类：

  • 机器无关优化 machine-independent optimization
    • 主要对中间表示 (IR) 进行改动，约等于中间表示优化 IR optimization
    • 如：死代码消除
  • 机器相关优化 machine-dependent optimization
    • 在生成目标代码 (汇编/机器码) 后针对特定 CPU 做优化，主要在目标代码上进行，所以约等于目标代码优化 target code optimization
    • 如：软流水、窥孔优化

而从优化范围看，又可分为：

  • 过程内优化 Intraprocedural，单个函数内，又可分为：
    • 局部优化 Local，基本块内的
      • 如：局部公共子表达式消除
    • 全局优化 Global，跨基本块、整个函数的
      • 如：死代码消除，常量传播
  • 过程间优化 Interprocedural，多个函数间
    • 如：函数内联替换

另外，我在语义分析时所做的是常量折叠，与常量传播不一样。

建立 CFG

即控制流图 control flow graph。

目前只有 IR 列表，没有 IR CFG，但不需要重建完整的 CFG，只需要建立一个轻量级的 CFG 骨架，记录各个基本块名和跳转关系，其他内容到原有 IR 中查即可。

基本块的最后一条语句只有 3 种可能 (如下)，ret 就结束了，jump、br 连到对应块上即可。

  • ret、ret <num>
  • jump <label>
  • br <value>, <label1>, <label2>

死代码消除

以代码块为单位的死代码消除做完了，现在开始以每条代码为单位来消除死代码。这比之前复杂了许多，块间、块内都要分析，也要在 CFG 上做迭代、用到真正的数据流分析算法了 (✧ω✧) ！

活跃变量分析

使用的活跃变量分析伪代码如下：

  • for ( each basic block B∈∪B\in \cupB∈∪ )
    • calculate defBdef_BdefB​、useBuse_BuseB​
  • IN[exit]=∅IN[exit] = \emptysetIN[exit]=∅
  • for ( each basic block B∈∪∖exitB\in \cup \setminus exitB∈∪∖exit )
    • IN[B]=∅IN[B] = \emptysetIN[B]=∅
  • while ( changes to any INININ occur )
    • for ( each basic block B∈∪∖exitB\in \cup \setminus exitB∈∪∖exit )
      • OUT[B]=⋃S∈successor(B)IN[S]OUT[B] = {\textstyle \bigcup_{S\in successor(B)}^{}} IN[S]OUT[B]=⋃S∈successor(B)​IN[S]
      • recalculate useBuse_BuseB​ with OUT[B]OUT[B]OUT[B]
      • IN[B]=useB∪(OUT[B]−defB)IN[B] = use_B \cup (OUT[B]-def_B)IN[B]=useB​∪(OUT[B]−defB​)

当 defBdef_BdefB​ 表示在 BBB 中被赋值的变量、useBuse_BuseB​ 表示在赋值前被使用的变量时，这样分析得到的 OUT[B]OUT[B]OUT[B] 就表示从 BBB 流出的数据流中，哪些变量的值在后面被直接用到。

OUT[B]OUT[B]OUT[B] 可以用来进行死代码消除，比如下面这段，假如 OUT[B]={a}OUT[B] = \left \{ a \right \}OUT[B]={a}，那 d = b 不会被用到可以去掉，a = b 下面又被赋值了也去掉，就简化成只剩 a = c 了。

糟糕的是，Koopa IR 远比这要复杂，除了局部变量，还有临时符号、局部数组、参数、全局变量；除了赋值，还有 load/store、ret %... (@_@)

我也不知道严谨的做法，只能凭感觉：

  1. 为什么把 defBdef_BdefB​ 从 OUT[B]OUT[B]OUT[B] 减去，因为 defBdef_BdefB​ 里的量已经在 BBB 中赋过值了，前面块的赋值没用了。

    而 Koopa IR 的 defBdef_BdefB​ 应该仅包含被赋值的局部变量、变量参数；数组的赋值并不一定覆盖前面的赋值，全局量的前面的赋值可能会被其他函数使用，所以 may analysis 认为都有用。

  2. 而加入的 useBuse_BuseB​ 原本表示在被赋值前为其他变量赋值的变量

    到了 Koopa IR 中将 useBuse_BuseB​ 扩展成在被赋值前产生影响的局部变量、变量参数和产生影响的局部数组(不要求赋值前)，这里产生影响是指：

    • 传给返回值
    • 为数组参数赋值
    • 为全局量赋值
    • 为函数传参
    • 影响控制流
    • 影响 OUT 中的局部变量、局部数组和变量参数 (这个需要动态更新)

    全局量和数组参数不用加入，因为后面死代码消除时会认为它们是始终会被用到的。

那临时符号怎么处理呢，我认为活跃变量分析时不用把临时符号加入 use / def / IN / OUT ，但需要用临时符号来传递影响。比如下面，是先记下 %0 影响了返回值，再由 @a_1 影响了 %0 得到 @a_1 对返回值产生了影响，临时符号在这里充当媒介。

之后的迭代算法不变。

这回真不知道对不对了，如果对了，我要狠狠奖励自己 (´∀｀)

消除死代码

什么样的代码是死代码，答：不会产生影响的代码

什么是产生影响，前面提过：

  • 传给返回值
  • 为数组参数赋值
  • 为全局量赋值
  • 为函数传参
  • 影响控制流
  • 再加一条：为 OUTOUTOUT 赋值
    • 局部数组的话所有赋值都有影响
    • 局部变量的话只有被用到的赋值才有影响，需要额外的块内分析

其他的，删！

修 Bug

Bug1，遇到当前块中 @a_1 没有用来产生影响且不在 OUT 中时，会把 alloc 误删。如果只是不删 alloc ，又会导致本该被删除的 @sce_result 留下来。所以增加新数据，统计所有块的 use，如果 alloc 的变量后面有 use 就不删了。

Bug2，还是 alloc 的问题。当某个局部变量、变量参数只在单个块中被使用时，它不会被放到 use 中，所以前面 alloc 处仍会被删除。那就从后往前遍历基本块，然后把只在块内使用的也记录下来。

Bug3，确认有用的指针时，没有考虑仅在当前块中有影响的数组，导致指令误删。

Bug4，这是个大问题！前面在求 use 时认为存入变量参数、局部变量、局部数组不算造成影响，因为它们不一定会在后面被用到，注意只是不一定；但如果它们在后面被用到了，前面将无法得知。

解决方法：每次重新计算 useBuse_BuseB​
