一处 shift/reduce

以下语法存在 shift/reduce conflict，还是空悬 else 相关的问题。

虽然 Lab6 中通过调整语法，解决了只有 if-else 时 else 不知道跟哪个 if 的问题，但是当 else 前面的两个 if 之间隔了 while ，又会出现二义性了。比如下面语法，有两种语法树。

这里 Bison 使用 LALR(1) 分析算法，通过前瞻看到的 token 来决定是将前瞻 token 读入 (移进)，还是将满足语法规则的符号替换 (归约)。

面对冲突时会默认选择移进，正好符合 else 就近匹配原则，所以虽然我忽略了这个问题这么久，它却没有造成什么影响。

调整如下，就是让 while 中出现 UnmatchedStmt 时，while 不能是 MatchedStmt 而要是 UnmatchedStmt，这样如果后面出现 else，只能被放入 while 中匹配。

初始化列表优化

说是优化，其实是之前偷懒欠的债 (-ω-) 。之前为了逻辑统一，会在语义分析时补上初始化列表中的所有值…

  • const int a[2][3] = {};

    -> const int a[2][3] = {{0,0,0},{0,0,0}};

  • int b[2][3] = {};

    -> int b[2][3] = {{0,0,0},{0,0,0}};

  • int c[2][3]; 这个倒不会补

然后在 IR 生成时：

  • 全局会把初始化列表写全
    • global @a_1 = alloc [[i32, 3], 2], {{0, 0, 0}, {0, 0, 0}}
    • global @b_1 = alloc [[i32, 3], 2], {{0, 0, 0}, {0, 0, 0}}
    • global @c_1 = alloc [[i32, 3], 2], {{0, 0, 0}, {0, 0, 0}}
  • 局部
    • @a_1 = alloc [[i32, 3], 2] 然后全赋值为 0
    • @b_1 = alloc [[i32, 3], 2] 然后全赋值为 0
    • @c_1 = alloc [[i32, 3], 2] 这个倒不会赋值

再生成 RISC-V：

  • 全局都是一列 .word

当出现 int a[20000] = {} 这种没有明确初始化的大数组时，会占用大量内存。虽然内存还够用，但为了给后面的优化算法留位置，我选择——改！

看下来，局部的逻辑太复杂，所以只改全局的。

语义分析时，全局的情况不用补 0

IR 生成时，全局只用 zeroinit

  • global @a_1 = alloc [[i32, 3], 2], zeroinit
  • global @b_1 = alloc [[i32, 3], 2], zeroinit
  • global @c_1 = alloc [[i32, 3], 2], zeroinit

RISC-V 生成时，全局都用 .zero 而不是一堆 .word

确实更快了，内存占用也少了。

总结

  • 增加了对函数声明的支持，这属于计划外的东西咯
  • 编了几个 SysY 项目，以好看的图图展示编译器的实力
  • 处理了一处 shift/reduce 冲突，以及优化了初始化列表

最让我高兴的是——初始化列表优化后，编译速度肉眼可见地提升了，内存占用也减少了 80% ，感觉好像找到了一点优化的感觉呢 (´∀｀)♡ 实际只是原来写的太烂了
