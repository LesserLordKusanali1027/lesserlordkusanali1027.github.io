语义分析

还是从符号表、常量折叠和报错检测三方面看。

符号表

之前符号表只需要记录 int常量-int值 和 int变量-string名 ，现在还要记录数组。不过数组长度和初始化内容不用记，只需记类型和名字：

  • int[]常量-string名
  • int[]变量-string名

此外，原来是通过内容是 int值 还是 string名 来区分 int 常量/变量两个类型的，但现在这四个类型不能再靠这种方法区分了，需要明确类型是 int/const int/int[]/const int[]，所以要修改符号表来支持这一点。

常量折叠

语义分析环节新增了一些需要计算表达式得出数字的地方，要把它们全部替换成单纯的数字，以便后续 IR 生成：

  • 数组定义时的大小，const int a[...]、int a[...]
  • 常量数组的初始化内容，const int a[] = {...}
  • 全局变量数组的初始化内容，int a[] = {...}

而这些地方不用算出数字来折叠：

  • 局部变量数组的初始化内容

  • 常量、变量数组的解引用，a[1]，即使是常量数组也不需要

    这也意味着数组解引用不能用于需要在语义分析环节算出的表达式中，比如：

    • 常量赋值 const int b = a[1] ×
    • 数组大小 int b[a[1]] ×

虽然手册没有对报错检测做强制要求，但认真实现一下还挺有趣的，表面上看错误的语义多种多样，但仔细想想也并非无法归纳出一套逻辑。

  • 常量数组不能在赋值语句中被赋值
  • int 不能用 {...} 初始化
  • int[] 不能用数字初始化

有符号名称相关的：

  • 虽然 C/C++ 支持数组与 int 同名，但这里认为数组名与 int 名不能重复

  • 数组解引用和变量同地位，值不会被记录到符号表中，不能给常量赋值…
  • …不能做数组大小…
  • …也不能给全局变量赋值。

关于 store 和 load

Koopa IR 中，当用比如 %1 = getelemptr ... 获取到指向某个数组元素的指针后，会跟着存储指令 store 1, %1 或者加载指令 %2 = load %1，指令中的临时符号 %1 应该被解释成指针值。

我之前犯了一个错误…

  • store 1, %1 -> li t0, 1; sw t0, ..(sp)
  • %2 = load %1 -> lw t0, ..(sp); sw t0, ..(sp)

…我把 %1 单纯当成了一个目标，但实际上它存的内容才是目标所在的地址，所以正确的转换是：

  • store 1, %1 -> li t0, 1; lw t1, ..(sp); sw t0, 0(t1)
  • %2 = load %1 -> lw t0, ..(sp); lw t0, 0(t0); sw t0, ..(sp)

看了一下，store 和 load 的这个位置只有两种情况，一种是 %1 临时变量存着指针值，另一种是 @ 存着变量，所以识别到 % 就用上面的格式替换，这是没问题哒。

思考一下，大概有以下内容：

  1. 初始化列表标准化的过程需要用到数组的维度信息，但这并不便集成到符号表中，所以要用新的状态变量来传递数组各个维度的大小

  1. 规则相关的——遇到数字时加入，遇到 { } 时判断是否对齐以及对齐到了哪个边界，{ } 结束时要知道如何补齐到对应边界

这些操作可以放到单独的一个类中，常量和变量不一样，所以设计两个类。

类需要有以下接口：

  1. 加入数组维度大小，比如 a[2][3]，就先加入 2 再加入 3
  2. 加入新的整数/AST，会返回 true/false 表示数字是被加入还是丢弃了；类内会随着整数的加入逐渐构造出 AST
  3. 告知遇到了 { }，需要得到返回 true/false 表示是否符合语义
  4. 告知 { } 结束，此时会自动补 0
  5. 要求返回新建的 AST

于是我用了一个栈的数组，里面提前放好一系列 ConstInitValList* ，当下层的 List 放满后，为它拼上 ConstInitVal，放入上层的 List，直到最上层的 List 放满，拼上 ConstInitVal 作为要返回的新 AST。

功能是正确的，设计也很好理解，但是代码、性能、内存等好像都很糟糕，以后需要重构。

  • getptr 是移动指针到某个元素处

两点

一，类型规定上需要用 load 把 **... 变成 *...

二，(俺寻思的) RISC-V 生成时，getelemptr @arr 用 addi；但 getptr @arr 不能用相同的逻辑，因为这两个 @arr 其实不一样。前者是 sp 加 @arr 偏移量是地址，但后者是 @arr 对应内存里的是地址 (你品，你细品)，后者反而更像 %... 该有的逻辑，所以干脆用 load 加载到 %... 中。
