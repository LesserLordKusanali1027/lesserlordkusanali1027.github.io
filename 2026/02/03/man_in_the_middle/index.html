Man-in-the-Middle

pwn.college 中Intercepting Communication模块最后一题Man-in-the-Middle的解题过程。题目地址

概述

参考了 pwn.college 之前的视频讲解，但视频里讲的是 IPv4 转发开启的情况，这里 IPv4 转发是关闭的，需要做些调整。

先看源代码，网络中有一个客户端10.0.0.2、一个服务器10.0.0.3以及我所在的10.0.0.1。

其中客户端和服务器之间进行着十分有序的交互。服务器部分代码如下，它做的事是验证密钥 -> 执行 echo/flag 命令。而客户端则是发送密钥 -> 发送命令。

具体的交互是这样：

  • 客户端 -> 服务器，发起 TCP 连接请求
  • 服务器 -> 客户端，接受请求，生成密钥，并等待接收密钥
  • 客户端 -> 服务器，从共享内存获得密钥，发送
  • 服务器 -> 客户端，密钥错误就结束连接，正确就等待接收命令
  • 客户端 -> 服务器，发送echo，再发送Hello, World!
  • 服务器 -> 客户端，返回Hello, World!
  • 结束

现在要做的就是介入两者的交互，将原本的命令echo Hello, World!篡改成flag，此即中间人攻击。

消息拦截

首先肯定要实现消息拦截（Intercept），不然我在 10.0.0.1 完全不知道客户端和服务器的交互，使用和前一题的相同的方法，ARP 投毒。

10.0.0.2 向 10.0.0.3 发消息时，需要知道 10.0.0.3 这个 IP 所对应的物理地址，然后由链路层向这个物理地址发送。如果伪造 ARP 响应报文将 10.0.0.3 关联到 10.0.0.1 的物理地址，我就可以收到这个消息了。
