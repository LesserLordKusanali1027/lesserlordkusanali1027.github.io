AST 数据结构 + Dump

语法规则

语法规则如下，包含有终结符和非终结符。

  • 非终结符：CompUnit，FuncDef，FuncType，Block，Stmt，Number
  • 终结符：IDENT，(，)，int，{，}，return，;，INT_CONST

语法分析就是将 token 流与语法规则不断匹配，寻找起始非终结符 CompUnit 要如何展开才能得到原本的 token 流，得到的展开方式以 AST 表示出来，是语法分析的目标。

比如之前的程序对应下面的 AST，中间节点都是非终结符，叶节点为终结符。

定义 AST 数据结构

为每个非终结符定义一个类，根据语法规则，在类中包含其他非终结符类和终结符作为元素。

由于这里语法规则较为简单，没有自己包含自己、多种语法选择等情况，所以数据结构很好写了。

要注意的地方：

  1. 使用智能指针 std::unique_ptr 而不是普通指针管理对象，可以在使用后自动释放内存，对于需要手动释放的情况减轻了负担。智能指针需要包含头文件 #include<memory>。
  2. 省略了几个终结符 (、)、{、}、return、;，这是因为目前语法比较简单，即使不存起来也很容易推出这些终结符的存在。
  3. 用基类 BaseAST + 各非终结符继承，便于统一接口。
  4. 成员用 std::unique_ptr<BaseAST> 而不是具体派生类，可能因为某个节点可能遇到多种选择，不能在定义时就确定 (虽然现在还没遇到)，在确定之前需要保持多态。

token 类型的传递

词法分析器在把 token 传递给语法分析器时，语法分析器需要知道 token 属于哪个类型 (如标识符/整数)，这里的 union 中有三个类型，标识符字符串、整数、AST 指针，但具体原理不知道。

补充：union 能在同一内存位置存放多种备选类型，节省空间。如果 union 中添加了带析构函数的类型，由于编译器还不知道 union 中是哪个类型，在不显式指定时，无法决定调用哪个析构函数，所以这里用了不带析构函数的类型。

token 种类声明

下面是 token，也是终结符种类的声明，在所有终结符中

  • int、return 作为保留字，有特定的内容，直接声明为特殊 token，不带类型
  • IDENT 是保留字之外的标识符字符串，声明为 <str_val> 类型
  • INT_CONST 是所有的整数，声明为 <int_val> 类型
  • (、)、{、}、;等剩下的字符，在 sysy.l 中直接以 . { return yytext[0]; } 返回原本符号，不需要用这些种类来表示了

注意：

  • 这里带类型的都是可能有不同值的，保留字的值固定所以不用带类型。
  • 这里的种类是与 sysy.l 词法分析返回的 token 种类对应的

从零开始的 IR_generator

在线文档中提供了两种生成 Koopa IR 的方法：

  1. 直接遍历 AST 输出文本 IR
  2. 定义 IR 数据结构，遍历 AST 来构建 IR，再输出文本

后者更利于 IR 优化，而且我觉得这样一层层的设计更加清晰，所以选择了后者。

现在要做的事情有：

  1. 定义 Koopa IR 数据结构
  2. 实现输出 Koopa IR 文本的函数
  3. 实现将 AST 转换成 Koopa IR 的函数

定义 Koopa IR 数据结构

在线文档中，IR 的结构是这样的：

  • Program 整个 Koopa IR 程序
    • 全局变量列表，含若干值 Value
      • Value 1
      • Value 2
      • …
    • 函数列表，含若干函数 Function
      • Function 1
        • 基本块列表，含若干基本块 BasicBlock
          • BasicBlock 1
            • 指令列表，含若干指令，也用 Value 表示
              • Value 1
              • Value 2
              • …
          • BasicBlock 2
          • …
      • Function 2
      • …

所以，定义如下类：

  1. 基类 BaseIR ，其他所有类均由此继承
  2. 程序类 ProgramIR ，包含全局变量列表和函数列表
    • 全局变量列表储存若干 ValueIR 指针，选择 std::vector<BaseIR*>
    • 函数列表储存若干 FunctionIR 指针，选择同上
  3. 函数类 FunctionIR ，包含基本块列表和函数名、返回值类型
  4. 基本块类 BasicBlockIR ，包含指令列表和基本块名
  5. 指令类 ValueIR ，包含指令操作码和操作数

注意的地方：

  • 全局变量和指令都用 ValueIR 表示，不确定之后是否会调整
  • 列表统一使用基类 <BaseIR*> ，暂时保持多态，是因为具体类型没必要在定义数据结构时就确定，在生成 Koopa IR 时确定就行
  • 之后可能会有多种指令的格式，ValueIR 的成员可能不固定，尚不知要怎么处理

实现 AST → IR 的构建函数

既然要遍历 AST 来构建 IR，那就需要找到 AST 结构和 IR 结构之间的对应关系，而 AST 的每个中间节点都是一个非终结符类，那么将 IR 类和非终结符类匹配，输入非终结符类来构建对应的 IR 类，是自然的想法。

于是：

  1. ProgramIR 根据 CompUnitAST 构建
  2. FunctionIR 根据 FuncDefAST 构建
  3. BasicBlockIR 根据 BlockAST 构建
  4. ValueIR 根据 StmtAST 构建
  5. 全局变量列表目前没有，暂时不用管
