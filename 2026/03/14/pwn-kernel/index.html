Kernel Security (内核安全)

发表于2026-03-14|Pwn.college
|浏览量:

Introduction

内核负责协调进程，管理外部资源。

(1) 外部资源举例：
比如一些只有内核可以控制的资源

  1. hlt指令使CPU暂停，等待外部中断来唤醒，只能由内核执行。
  2. in、out指令与外设交互。
  3. 特殊寄存器，cr3寄存器用于将虚拟地址<->物理地址，只能在内核态下修改(可用mov)；MSR_LSTAR寄存器用于确定syscall跳转地址，在内核中使用wrmsr、rdmsr来写、读。

(2) x86的权限等级(Rings)：

分成Ring3~Ring0四个等级，但一般只使用Ring3(用户态)和Ring0(内核态)，Ring2、Ring1只有在需要提供一部分但非全部的内核权限时启用。

虚拟机出现后，曾使用Ring1，因为虚拟机内核需要一部分权限，但不是所有权限。这导致了一些混乱，所以现在虚拟机工作在Ring -1(Hypervisor Mode)下，一种特殊的专用于虚拟机的特权模式。

提到了单体内核架构(Linux)、微内核架构(Minux)和混合内核架构(MacOS)。单体内核架构下，程序需要内核权限时，会进入内核执行，这时程序的漏洞就成了内核漏洞。

(3) x86如何切换Rings：
整体看一下

  1. 系统启动时，内核在Ring0下将MSR_LSTAR指向一个被称作enter syscall的例程。
  2. 用户空间(Ring3)中的进程调用syscall的过程：
    • 进程权限置为Ring0
    • 程序流跳转到MSR_LSTAR指向的地址
    • 返回地址存入rcx(没有放在栈上，因为栈可能被其他线程干扰)
    • 结束后，还会调用sysret返回用户空间，跳转到rcx指向的地址

(4) 内核和用户的虚拟地址空间：

内核与用户其实共用一块虚拟地址空间，但内核在高地址，用户在低地址。之前用户进程地址的高2字节都是0，这是因为更高的地址被分给了内核，来存放内核的数据和代码。调用syscall时会跳转到更高的内核地址去执行这些代码，但不能在Ring3下自己跳转(权限不够)。

本质上，内核和用户没什么区别，都是一堆代码和数据在运行，只是权限等级不同，所以程序的漏洞在内核中也能看到。

(5) 内核漏洞利用：
通过网络数据包、用户态进程、外设等途径，可以实现沙箱逃逸、植入Rootkits等攻击。

Kernel Modules

(1) Kernel module 是什么

用户态下，程序加载.so文件作为库（如libc.so.6），内核态下，Kernel module .ko被加载作为库，与内核程序一起在内核权限下运行。

内核模块可被用来：

  • 作为设备驱动
  • 管理文件系统
  • 网络功能

(2) 我们是如何与 Kernel module 交互的

  • system calls

    过去有 syscall handler 可以让 Kernel module 拦截原本某个 syscall ，替换成其他功能；现在该功能被废除，改用Root Kits实现这点。

  • interrupts

    可用LIDT和LGDT指令挂载中断处理程序，调用int 42来触发。

    还有int3，会触发 SIGTRAP 中断，但可以被 hook。

  • files

    文件是与内核模块交互的最常用的途径。

    • /dev：用来放置设备文件。
    • /proc：开始是用来获取运行进程信息的，后来被用于内核模块交互。
    • /sys：原本在/proc中的文件过于混乱，部分被移动到/sys中。

    用户态下可以用open()打开这些文件，然后通过文件与内核模块交互：

    • 使用read()、write()读写，一般用于操作 stream data，如获取摄像头视频。
    • 使用ioctl()，一般用于传输 non-stream data，如设置摄像头参数。

(3) Kernel Modules 做了什么

Kernel Module 什么都可以做，因为拥有最高权限，但普遍来说：

  • 从用户态 read data，用copy_from_user
  • 干活，读写文件、与硬件交互等
  • 向用户态 write data，用copy_to_user
  • 返回 userspace

Escaping Seccomp

这是一种内核限制 syscall 的沙箱技术，但如果能用允许的 syscall 触发内核漏洞，可实现沙箱逃逸。

(1) Seccomp 是如何工作的

thread_info中的flags的某一位是TIF_SECCOMP，表示是否启用 seccomp 。

Seccomp 执行前，会通过secure_computing检查，如果TIF_SECCOMP被设置了，就执行 seccomp 过滤器。

(2) 逃离 seccomp

将当前进程的flags中TIF_SECCOMP对应 bit 清0即可。

那如何找到当前进程的task_struct：内核会一直将寄存器gs指向当前进程，于是可以：

  • 通过gs->thread_info.flags找到当前进程的 flags
  • 将TIF_SECCOMP位清0
  • 逃出 seccomp

注意：当前进程逃出 seccomp 后，其子进程并不会逃出，因为 seccomp 继承信息储存在其他地方。

(3) 两阶段 escaping

编译时带上-lseccomp，因为 seccomp 属于系统库，需要用-l<库名>显式指定。

(1) The Page Table

最初使用 Page Table ，将物理内存分割成许多 Page ，将 Page 编号和物理内存地址的对应关系储存在 Page Table 中。

后来为了能操作更大的物理内存，发展出了二级 Page Table。

之后 RAM 越来越大，Page Table 的级数越来越多。

那么比如4级，每个 Page 大小为0x1000字节对应12位，每个 Page Table 有512项对应9位。

所以总共可以容纳48位的地址。

而汇编中的操作mov rax, [rbx]，从物理地址获取就要经过4次查表。

(2) 进程间的独立

不同进程使用不同的物理内存，也有不同的PML4，PML4的物理地址会被加载到cr3中。

(3) 虚拟机间的独立

如果是虚拟机呢，如何让多个内核独立，如何让每个虚拟机都认为自己可以管理整个物理内存。

使用了扩展页表Extended Page Table，宿主机每次都用EPT将虚拟机认为的物理地址转换成真实的物理地址。这个过程看起来很漫长：

(4) 两个宏

内核有两个提供地址映射功能的宏：

  • phys_to_virt()将物理地址 -> 内核虚拟地址。
  • virt_to_phys()将内核虚拟地址 -> 物理地址。

熟悉的场景：

  • canary保护 kernel 栈
  • kASLR将 kernel 加载到随机地址
  • 堆栈NX，可用ROP

Writing Kernel Shellcode

(1) 从用户到内核

用户空间中可运行syscall，但如果在内核执行syscall，会崩溃，因为syscall默认在用户空间被执行。

那要怎么做：调用内核提供的 API函数。

  • 想提权，调用commit_creds(prepare_kernel_cred(0))
  • 想越狱，调用task_struct->thread_info.flags &= ~TIF_SECCOMP
  • 想执行命令，调用run_cmd("/path/to/your/cmd")
  • 不需要syscall，只需找到task_struct等数据结构所在地址，以及 call API函数(的地址)。

(2) 寻找API函数所在地址

  • kASLR 关闭，可以看/proc/kallsyms，可以看vmlinux
  • kASLR 开启，泄露某个地址后算偏移，就像 libc 一样

(3) 寻找 task struct 地址

内核将段寄存器(segment register)gs指向当前进程的 task struct 。
