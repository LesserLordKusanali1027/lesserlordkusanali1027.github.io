短路求值

实现

短路求值，是 C/C++ 中一个很熟悉的概念，这里的含义也一样，就是指在计算含有逻辑运算符 || 和 && 的算式时，当参与计算的量满足某些条件，可以提前出结果。一般计算顺序从左向右，有两种情况：

  • a || b || ... 中，当左边出现非 0 时，提前得到结果为 1
  • a && b && ... 中，当左边出现 0 时，提前得到结果为 0

借助分支和跳转指令是可以实现的，但要想下如何排布这些指令才能达到目的，我的设计如下，以 a && b && c 格式为例。

对上图的说明：

  1. 五个基本块，序号是它们的生成顺序。
  2. 每个子表达式 exp<num> 对应一个基本块，这里 exp1、exp2、exp3 各对应基本块 1、2、3 ，基本块内有表达式计算和 br 指令 两部分。
  3. br 指令用于短路求值跳转，这里当结果 a/b 为 0 时，br 就会跳转到短路求值块 and_sce_1 中，将结果提前赋值为 0
  4. 如果不符合短路求值条件，就会一直沿着 sub_exp 执行，直到 sub_exp_2 算出结果后跳转到结束块
  5. and_exit_1 为结束块，不论是经过短路求值还是正常算完，最后都会跳到这里，继续后续计算

基本块命名的约定：

  • &&、|| 连接的每个式子对应一个基本块 sub_exp；短路求值基本块为 and_sce 和 or_sce；结束块为 and_exit 和 or_exit
  • 有了块名的约定，前面的 br、jump 指令就可以提前知晓跳转目标，不需要等到目标块创建后再回填。代价嘛，就是要保存的状态量增加了 5 个

细节：上例中最后结果 c 的求出有两条路径：

  1. 全部算完得到结果 c

  2. 短路求值得到 c

    那么，如果 c 是保存在临时符号中的，IR 代码中会有两处为 %c 赋值的地方，这是违反 Koopa IR 的单赋值性质的，所以要用变量来存放结果了。这里约定结果存放在变量 @sce_result 中，直接在 IR 生成中添加即可。

    PS：因为其他的变量都在语义分析中被添加了编号，所以不用担心变量名冲突。

  2. 对于 || 短路求值
    • 操作部分在 LOrExpAST_2 中生成，善后部分在 ExpAST 中生成
    • 善后时同样要区分 LOrExpAST_1 和 LOrExpAST_2，只需要给 LOrExpAST_2 善后

总结

关键内容：

  • 实现了对 if-else 分支指令的处理
    • 修改了原本语法规则来解决空悬 else 问题
    • IR 生成中，确定了负责生成 then、else、end 基本块的节点
    • IR 生成使用了在 ret 后添新建标签的无脑方法，解决了 return 后还有指令时带来的 bug
  • 实现了含有 &&、|| 的表达式的短路求值
    • IR 生成中，设计了一套排布 IR 代码的结构，借助 br 和 jump
    • 模块化做得不错 (试图用专业术语来增加逼格)，其他节点不需要关注短路求值结构内的细节
    • 职责分离做得也不错，对于不含 && 和 || 的表达式，短路求值不会启用

程序流不再是线性，IR 基本块从原来的一个变成了好多个，一些节点需要处理的内容大大增加。

IR生成中，我逐渐感觉到递归的有趣…写单个节点时，我并不知道最后会写成什么样，而全写完开始测试了，就如同开奖前的瞬间，我才会成功->欢呼，或者败北->debug。

一点点写递归的感悟：要有耐心，先自顶向下设计，再自底向上实现。
