…各个模块已经齐了，Lab3 要在这些文件中增加内容，使原来只能放数字的 return <num>; 变成可以放一个表达式的 return <exp>;

而为了管理好逐渐增加的内容，我也重构了几个文件，改了改各个层之间的接口，这还挺花时间的。至于具体花了多少时间，可以看下距离我做完 Lab2 过去了多久 ( ᗜ ‸ ᗜ )

IR 生成

做了两件事：改接口、增加新的 IR 生成逻辑。

改接口

为什么要改接口呢 (*´･д･)?

Lab1 中，为了遍历 AST 树，为每种 AST 结点设计了一个接口函数 (如下)。

这是因为 parser.hpp 把 AST 成员都定义成了 std::unique_ptr<BaseAST> 类型，所以传入接口前要用 dynamic_cast 将一堆 BaseAST 进行显式类型转换，这很丑陋。随着类型增加，需要用 dynamic_cast 判断的类型越来越多，且这些判断难以提炼出统一的格式。

回到设计接口的出发点，想对不同 AST 派生类执行不同处理，有 3 种方法，选择第 3 种吧。

  1. 直接在 AST 类内实现输出 IR 的函数。不符合分层，早就被 pass 了。
  2. 外部接口 + dynamic_cast。Lab1 用的，需要手动判断类型，很麻烦。dynamic_cast 更适合用来保险，而不是实现功能。
  3. 外部接口类 + Visitor 模式。Visitor 模式的神奇之处就是可以根据运行时类型自动选择接口函数，不需要显式声明，也统一了接口类型。

Visitor 模式部分实现：

  • parser.hpp
  • koopa.hpp，这里在 Visitor 类中除接口函数外还定义了其他变量，用来在不同接口间相互调用时传递信息。

新的 IR 生成逻辑

先前只有一条 IR 所以直接替换就行，现在有了多条 IR (如下)，不同 IR 语句间要有临时符号的传递。同时 AST 也更复杂了，无法像先前一样直接用 value_ir -> operand = fd -> num; 就能获取到数值。

于是，在 Visitor_ast 类中定义一个栈，来辅助进行临时符号和数值的传递，这就是新增的 IR 逻辑。

a < b c = lt a, b
a >= b c = ge a, b
a <= b c = le a, b
a != b c = ne a, b
a == b c = eq a, b
a && b c = ne a, 0; d = ne b, 0; e = and c, d
a || b c = ne a, 0; d = ne b, 0; e = or c, d

总结

关键内容：

  • 选择每个规则一个 AST
  • Visitor 模式实现对不同派生类调用不同接口函数
  • Visitor 类中添加成员实现不同接口间数据传递
  • IR 生成时用栈传递临时符号和数值
  • RISC-V 生成实现了无脑的寄存器重用，定义工具函数进行了重构

和 Lab2 相比，从 AST 生成 IR 的函数美观了许多，转换过程也从简单的换格式变成了有栈参与的表达式计算过程，这也让我感觉到 IR 确实是有用的。

然而 IR 的数据结构仍然过于简单，看起来难以支持更复杂的 AST；从 IR 到 RISC-V 的过程也依然是简单的替换。

从 AST 到 IR 再到 RISC-V，这一系列步骤的作用，还有待在后面的 Lab 中继续探索。
